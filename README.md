# ai-flow

**ai-flow** — это готовый «рабочий процесс» (workflow) для разработки проектов с помощью ИИ-агентов.
Он превращает свободное «сделай мне фичу» в предсказуемый конвейер:

> **идея → PRD → спецификация → задачи → автоматическое выполнение агентом → самоподдержка знаний и документации**

Вся инфраструктура лежит в одном каталоге — `ai-flow/`. В корне проекта остаётся только то, что
ИИ-инструменты находят автоматически: `README.md`, `CLAUDE.md`, `AGENTS.md`, `.claude/`, `.serena/`.

> Этот README — **учебник**. Если вы впервые видите ИИ-агентов, читайте подряд: разделы
> [«Зачем это нужно»](#зачем-это-нужно), [«Словарь»](#словарь-основные-понятия) и
> [«С чего начать»](#с-чего-начать-пошагово) дадут рабочее понимание за 15 минут. Опытным — сразу к
> [«Командам»](#справочник-команд) и [«Как устроен оркестратор»](#как-устроен-оркестратор-под-капотом).

## Зачем это нужно

Если просто попросить ИИ «напиши мне приложение», он ответит правдоподобным кодом, но:

- **забудет договорённости** — на следующей задаче он не помнит, как вы называете файлы, где лежит
  бизнес-логика, какими командами собирается проект;
- **сам придумает недостающее** — вместо того чтобы спросить, он «догадается», и часто неверно;
- **не оставит следов** — через неделю ни вы, ни другой агент не поймёте, что и зачем было сделано.

ai-flow решает это тремя приёмами:

1. **Разделяй и властвуй.** Большая цель дробится на маленькие самодостаточные *задачи*
   (100–500 строк кода каждая), которые агент делает по одной. Маленькая задача с полным описанием
   выполняется надёжнее, чем большая расплывчатая.
2. **Память вместо угадывания.** Правила проекта, соглашения и паттерны хранятся в файлах
   (`CLAUDE.md` + *память Serena*) и **автоматически подкладываются агенту** в каждую задачу. Он не
   гадает — он читает.
3. **Документация обновляется сама.** После каждой задачи агент дописывает журнал изменений, обновляет
   спецификацию фичи и — если появился переиспользуемый паттерн — сохраняет его в память. Знания не
   теряются.

Итог: вы описываете, что хотите, один раз и на понятном языке, а дальше конвейер выполняет задачи
пачкой, коммитит результат и сам поддерживает документацию в актуальном состоянии.

## Словарь: основные понятия

Прежде чем читать дальше — короткий словарь. Эти слова встречаются во всём README.

| Термин | Что это простыми словами |
|---|---|
| **Агент / кодинг-CLI** | Программа-исполнитель на базе LLM, которая умеет читать и писать файлы в вашем проекте из командной строки. Например, [Claude Code](https://docs.claude.com/claude-code), OpenAI Codex CLI, zcode/GLM. Именно агент пишет код по задаче. |
| **Оркестратор** | Скрипт `ai-flow/run_tasks.py`. Он находит задачи, по очереди запускает агента на каждой, ждёт результат, коммитит и переносит выполненную задачу в архив. «Дирижёр» для агентов. |
| **Спецификация (спека)** | Документ в `ai-flow/docs/specs/`, описывающий, **что** должно быть построено: интерфейсы, структуры данных, критерии приёмки. Read-only контекст для последующих шагов. |
| **Задача (task)** | Один файл-инструкция в `ai-flow/docs/tasks/<фича>/`. Самодостаточное описание одного маленького изменения: что менять, в каких файлах, как проверить готовность. |
| **Фича-папка** | Папка `ai-flow/docs/tasks/<YYYYMMddHHmm_FEATURE>/`, куда сгруппированы все задачи одной доработки + `README.md` с её обзором (DesignReview) + подпапка `done/` для выполненных. |
| **Память (Serena memory)** | Обычные `.md`-файлы в `.serena/memories/` с правилами и паттернами проекта («как мы пишем тесты», «где лежит доступ к БД»). Читаются агентом по мере надобности. [Serena](https://github.com/oraios/serena) — инструмент, дающий агенту удобный поиск по коду и памяти. |
| **CLAUDE.md** | Единый свод правил проекта в корне. Главный источник истины: агенты любых инструментов обязаны его читать (`AGENTS.md` просто перенаправляет на него). |
| **PRD** | *Product Requirements Document* — первичный документ нового проекта: видение, пользователи, пользовательские сценарии, границы функциональности, ограничения и направление архитектуры. Пишется **до** спеки, через интервью (промт `PROMT_PRD` / скилл `/new-prd`), и ложится в `ai-flow/docs/specs/PRD.md`. |
| **Промт (prompt)** | Шаблон-инструкция в `ai-flow/docs/prompts/`. Их пять — для дискавери и написания PRD нового проекта, написания спеки, декомпозиции на задачи, инструкций агенту-исполнителю и наполнения памяти. |
| **Хук (hook)** | Небольшой скрипт, который среда запускает автоматически на событие (здесь — при старте сессии). Единственное настоящее «автоматически» в этом наборе. |
| **Субагент / скилл** | Специализированные помощники Claude Code. Субагенты `prd-author` (интервьюирует и пишет PRD), `task-author` (создаёт задачи) и `doc-keeper` (ведёт доки и память); скиллы `/new-prd`, `/new-task` и `/sync-docs` — удобные точки входа к ним. |
| **Маркер завершения** | Строка `<promise>COMPLETE</promise>`, которую агент печатает, закончив задачу. По ней оркестратор понимает, что можно коммитить и двигаться дальше. |

## Как это работает — общая картина

```
              ВЫ описываете идею нового проекта
                          │
                          ▼
                     PROMT_PRD                ← только для проекта с нуля (/new-prd)
                  (интервью → PRD:
                   видение, пользователи,
                   сценарии, ограничения)
                          │
                          ▼
               ai-flow/docs/specs/PRD.md
                          │
                 ВЫ описываете, что хотите (или берёте это из PRD)
                          │
        ┌─────────────────┼──────────────────────────┐
        ▼                 ▼                            ▼
  PROMT_SERENA        PROMT_SPEC                  PROMT_TASKS
  (наполнить          (написать                  (разбить на
   память проекта)     спецификацию)              атомарные задачи)
        │                 │                            │
        ▼                 ▼                            ▼
 .serena/memories/   ai-flow/docs/specs/    ai-flow/docs/tasks/<фича>/
   *.md + таблица      README + NN-*.md         README (DesignReview)
   в CLAUDE.md                                  + файлы задач
        └───────────┬─────┴───────────────┬──────────┘
                    ▼                      ▼
            python ai-flow/run_tasks.py  (ОРКЕСТРАТОР)
                    │
                    │  для каждой готовой задачи:
                    │   1. собирает промт: PROMT_AGENT + CLAUDE.md +
                    │      указатели на спеки/память + DesignReview +
                    │      журнал + текст задачи
                    │   2. запускает АГЕНТА (claude / codex / zcode)
                    │   3. агент пишет код, тесты, обновляет доки/память,
                    │      печатает <promise>COMPLETE</promise>
                    │   4. оркестратор переносит задачу в done/ и коммитит
                    ▼
        Готовый код + обновлённые:
          docs/CHANGELOG.md · docs/specs/<фича>/{README,IMPLEMENTED}.md
          · .serena/memories/* · git-история
```

Ключевая идея: **оркестратор не думает** — он лишь подкладывает агенту весь нужный контекст и следит
за порядком. Всё «мышление» происходит внутри агента, но в жёстких рамках задачи и правил проекта.

## Что нужно перед стартом (prerequisites)

| Нужно | Зачем | Как проверить/поставить |
|---|---|---|
| **Python 3.10+** | На нём написаны `run_tasks.py` и `init.py`. | `python --version` |
| **PyYAML** | Оркестратор читает конфиг `agents.yml`. | `pip install pyyaml` |
| **Кодинг-CLI агента** | Собственно исполнитель кода. По умолчанию — **Claude Code**. | Установите и авторизуйтесь: [Claude Code](https://docs.claude.com/claude-code). Альтернативы — Codex CLI, zcode. |
| **git** (желательно) | Для авто-коммитов после каждой задачи. Без него оркестратор просто пропустит коммит. | `git --version` |
| **uv / uvx** (желательно) | Запускает сервер памяти [Serena](https://github.com/oraios/serena). | `pip install uv` или [astral.sh/uv](https://astral.sh/uv) |

> **Минимум, чтобы попробовать:** Python + PyYAML + установленный Claude Code. Serena и git можно
> добавить позже — флоу деградирует мягко (память читается как обычные `.md`, коммиты просто не делаются).

## С чего начать (пошагово)

Есть два сценария. Выберите свой.

### Сценарий A. Начать новый проект на базе шаблона (проще всего)

Этот репозиторий **сам является** шаблоном — вся инфраструктура уже на месте. Достаточно взять его за
основу своего проекта:

```bash
# 1. Возьмите шаблон как основу нового проекта (клонирование + отвязка от истории шаблона)
git clone <url-этого-репозитория> my-project
cd my-project
rm -rf .git && git init          # начать свою git-историю (по желанию)

# 2. Поставьте зависимость оркестратора
pip install pyyaml

# 3. Настройте Serena под ваш инструмент (создаст MCP-подключение)
python ai-flow/init.py setup-serena --tool claude

# 4. Онбординг: пусть агент изучит проект и наполнит память + обзор в specs/README
#    — в Claude Code выполните скилл:
/sync-docs
#    (или вручную скормите агенту ai-flow/docs/prompts/PROMT_SERENA.md)
```

### Сценарий B. Развернуть флоу в отдельную папку / существующий проект

Используйте установщик `init.py` — он копирует все нужные файлы в целевую папку и адаптирует их:

```bash
# новый проект: язык кода + язык общения агента с вами
python ai-flow/init.py init --target ../my-project --tool claude --lang typescript --comm-lang ru
```

`init` не перезаписывает уже существующие файлы (если не передать `--force`), а `.gitignore` — дополняет.
Подробнее про флаги — в разделе [«Справочник команд»](#справочник-команд).

### Дальше (общее для A и B) — цикл разработки

```bash
# 5. Опишите доработку и разбейте её на задачи (скилл вызывает субагента task-author):
/new-task Добавить вход по email и паролю с выдачей JWT

# 6. Запустите выполнение — агент сделает задачи по очереди:
python ai-flow/run_tasks.py

# 7. Периодически освежайте память/доки, если появились новые паттерны:
/sync-docs
```

> `/new-task` и `/sync-docs` — это **скиллы Claude Code** (набираются в его чате). Если работаете не
> в Claude Code, вместо скиллов вручную скармливайте агенту соответствующий промт из
> `ai-flow/docs/prompts/` — результат тот же.

## Новый проект с нуля: сначала PRD

Если проект **greenfield** (есть идея, но ещё нет ни кода, ни спеки), начните не с задачи, а с
**дискавери** — сбора первичных требований. Для этого есть отдельный агент `prd-author` и скилл
`/new-prd`. Он **проводит интервью**, задавая вопросы по темам, и по ответам пишет
`ai-flow/docs/specs/PRD.md`.

```bash
# в чате Claude Code — запустите интервью (в скобках можно сразу дать грубую идею):
/new-prd Мобильное приложение для учёта личных расходов с распознаванием чеков
```

Агент спросит вас **по секциям** (не вываливая всё сразу) и уточнит:

| Тема | Что выясняет |
|---|---|
| Видение и проблема | что за продукт в одном предложении, чью боль решает и почему сейчас |
| Пользователи и персоны | кто основные/второстепенные пользователи, их контекст и уровень |
| Пользовательские сценарии | 3–7 ключевых сценариев «триггер → шаги → результат»; какой главный |
| Функциональность | возможности, разбитые на **MVP** и «потом»; что явно НЕ входит в v1 |
| Нефункциональные требования и ограничения | производительность, масштаб, безопасность, приватность, комплаенс, бюджет, сроки |
| Направление архитектуры | платформа, стек (или разумный дефолт как допущение), деплой, монолит/сервисы |
| Модель данных | ключевые сущности и связи, источники и чувствительность данных |
| Интеграции | сторонние API/сервисы, авторизация, платежи; жёсткие зависимости vs опциональные |
| Метрики успеха | измеримые сигналы, что продукт работает и полезен |
| Допущения и риски | явные допущения, главные технические/продуктовые/срочные риски |

Правило то же, что и во всём флоу: **агент не выдумывает** — каждое неизвестное становится либо
*Open Question*, либо явным *Assumption* с обоснованием. Готовый `PRD.md` содержит секции: видение,
проблема, цели и метрики, персоны, сценарии, функциональные требования (MVP/потом), нефункциональные
требования, архитектурное направление, модель данных, интеграции, допущения и риски, вне области,
открытые вопросы и **Next Steps**.

**Дальше по PRD:**

```bash
# 1. Превратить области PRD в детальные спеки (промт PROMT_SPEC — вручную скормите его агенту):
#    вход: PRD.md → выход: ai-flow/docs/specs/NN-*.md
# 2. Онбординг новой кодовой базы и наполнение памяти:
/sync-docs
# 3. Декомпозировать первую фичу в задачи:
/new-task <первая фича из PRD>
```

> Интервью **интерактивно**, поэтому, в отличие от `/new-task` и `/sync-docs`, `/new-prd` ведёт диалог
> прямо в основном чате (запущенный субагент не может задавать вам вопросы). Вне Claude Code скормите
> агенту `ai-flow/docs/prompts/PROMT_PRD.md` в его собственной сессии — результат тот же.

## Сквозной пример: фича «вход по логину»

Покажем весь путь на конкретной доработке. Предположим, проект уже онбординжен (шаг 4 выше выполнен).

**1. Описываем и декомпозируем.** В чате Claude Code:

```
/new-task Вход по email+паролю: эндпоинт POST /login, проверка пароля (bcrypt),
выдача JWT на 24 часа, негативные сценарии на неверные креды.
```

Субагент `task-author` прочитает правила проекта и память, возьмёт формат из `PROMT_TASKS.md` и создаст:

```
ai-flow/docs/tasks/202607131500_user-login/
├── README.md                              # DesignReview: суть, ценность, архитектура, критерии
├── 202607131500_add-login-endpoint.md     # задача 1: эндпоинт + DTO
├── 202607131505_verify-password.md         # задача 2: проверка пароля (Depends on: add-login-endpoint)
└── 202607131510_issue-jwt.md               # задача 3: выдача JWT     (Depends on: verify-password)
```

**2. (Опционально) смотрим план без запуска:**

```bash
python ai-flow/run_tasks.py --feature user-login --dry-run
```

**3. Запускаем выполнение только этой фичи:**

```bash
python ai-flow/run_tasks.py --feature user-login
```

Оркестратор для каждой задачи по очереди (учитывая `Depends on:`):

- соберёт агенту промт (правила, память, DesignReview фичи, журнал, текст задачи);
- запустит агента — тот прочитает существующий код, напишет реализацию и тесты, соберёт проект,
  допишет `CHANGELOG.md`, обновит `specs/202607131500_user-login/{README,IMPLEMENTED}.md`;
- увидев `<promise>COMPLETE</promise>`, **перенесёт** файл задачи в `202607131500_user-login/done/`
  и сделает коммит `feat: task #202607131500_add-login-endpoint - Add login endpoint`.

**4. Что осталось после прогона:**

- рабочий код + тесты в проекте;
- `ai-flow/docs/tasks/202607131500_user-login/done/` — все три задачи;
- `ai-flow/docs/specs/202607131500_user-login/README.md` (цель) и `IMPLEMENTED.md` (как сделано);
- дописанный `ai-flow/docs/CHANGELOG.md`;
- при появлении паттерна (напр., «как мы валидируем DTO») — новая запись в `.serena/memories/` и
  строка в таблице `CLAUDE.md`;
- три коммита в git.

Если во время работы всплыл общий паттерн, освежите знания: `/sync-docs`.

## Структура репозитория

```
<repo>/
├── README.md              # этот файл — как пользоваться флоу
├── CLAUDE.md              # ЕДИНЫЙ свод правил + таблица read_memory(...) (на English)
├── AGENTS.md              # редирект прочих инструментов (Codex/Cursor/…) на CLAUDE.md
├── .claude/
│   ├── settings.json      # сюда init.py вписывает SessionStart-хук
│   ├── agents/            # субагенты: prd-author (PRD), task-author (задачи), doc-keeper (доки/память)
│   └── skills/            # скиллы: new-prd, new-task, sync-docs (тонкие обёртки над субагентами)
├── .serena/
│   ├── project.yml        # конфиг Serena (язык проекта задаётся здесь)
│   └── memories/          # ПАМЯТЬ проекта — обычные .md (suggested-commands, task-completion, …)
└── ai-flow/               # ВСЯ инфраструктура флоу
    ├── run_tasks.py       # ОРКЕСТРАТОР авто-выполнения задач
    ├── init.py            # УСТАНОВЩИК: развернуть/адаптировать под инструмент + настроить Serena
    ├── agents.yml         # конфиг агентов (claude / codex / zcode) и оркестратора
    ├── hooks/
    │   ├── check_memory_sync.py    # хук: сверяет таблицу в CLAUDE.md с файлами памяти
    │   └── hooks.config.json       # настройки хука
    └── docs/
        ├── prompts/       # 5 промтов: PROMT_PRD / PROMT_SPEC / PROMT_TASKS / PROMT_AGENT / PROMT_SERENA
        ├── specs/         # спеки + функциональность: root README + <feature>/{README,IMPLEMENTED}
        ├── tasks/         # задачи по фичам: <YYYYMMddHHmm_FEATURE>/{README, задачи, done/}
        └── CHANGELOG.md   # хронологический журнал выполненных задач
```

**Как читать эту структуру.** Всё, что нужно инструментам «из коробки» (правила и авто-обнаружение),
живёт в корне и в `.claude/`/`.serena/`. Всё, что относится к самому процессу (скрипты, промты, задачи,
спеки, журнал) — под `ai-flow/`. Такое разделение позволяет [не тащить флоу в git](#не-коммитить-флоу-в-git),
если он вам нужен только локально.

### Роли пяти промтов

| Промт | Вход | Выход | Когда запускать |
|---|---|---|---|
| `PROMT_PRD.md` | идея нового проекта | PRD `specs/PRD.md` (через интервью) | **старт проекта с нуля** (`/new-prd`) |
| `PROMT_SERENA.md` | код проекта | память `.serena/memories/*` + таблица в `CLAUDE.md` + обзор в `specs/README` | **онбординг** и когда доки отстали (`/sync-docs`) |
| `PROMT_SPEC.md` | требования | файлы спеки `specs/NN-*.md` | когда нужна детальная спецификация перед декомпозицией |
| `PROMT_TASKS.md` | спека/требования | фича-папка с DesignReview + задачами | на каждую новую доработку (`/new-task`) |
| `PROMT_AGENT.md` | одна задача (+ контекст) | реализованный код | **автоматически** — его использует оркестратор, вручную не запускают |

## Справочник команд

### Оркестратор — `run_tasks.py`

Выполняет готовые задачи по очереди. Требует `pip install pyyaml`.

```bash
python ai-flow/run_tasks.py                      # агент по умолчанию из agents.yml (claude/sonnet)
python ai-flow/run_tasks.py --agent codex --model gpt-5-codex   # другой агент/модель
python ai-flow/run_tasks.py --agent zcode --model glm-4.6
python ai-flow/run_tasks.py --feature user-login # только задачи из папок фич, где имя совпадает
python ai-flow/run_tasks.py --task add-login     # только задачи, чьё имя содержит подстроку
python ai-flow/run_tasks.py --dry-run            # показать план, ничего не выполняя
python ai-flow/run_tasks.py --config path.yml    # свой конфиг вместо ai-flow/agents.yml
```

| Флаг | Что делает |
|---|---|
| `--agent NAME` | имя агента из `agents.yml` (перекрывает `default_agent`) |
| `--model M` | перекрывает модель выбранного агента |
| `--feature NAME` | запускать только задачи из фича-папок, где `NAME` — подстрока имени |
| `--task NAME` | запускать только задачи, где `NAME` — подстрока имени файла; после одной задачи стоп |
| `--dry-run` | вывести, что и в каком порядке запустилось бы, без запуска агента |
| `--config PATH` | путь к альтернативному `agents.yml` |

### Установщик — `init.py`

Разворачивает флоу в целевую папку и адаптирует под конкретный инструмент.

```bash
# развернуть в новую папку: язык кода + язык общения
python ai-flow/init.py init --target ../my-project --tool claude --lang typescript --comm-lang ru

# развернуть в текущую папку под Codex
python ai-flow/init.py init --tool codex

# только пере-сгенерировать адаптеры субагентов/скиллов под другой инструмент
python ai-flow/init.py adapt --tool cursor

# только настроить Serena MCP под инструмент
python ai-flow/init.py setup-serena --tool claude

# список поддерживаемых инструментов
python ai-flow/init.py list-tools
```

Флаги `init`:

| Флаг | По умолчанию | Смысл |
|---|---|---|
| `--target DIR` | `.` | куда разворачивать |
| `--tool TOOL` | `claude` | целевой инструмент (`claude`/`codex`/`cursor`/`gemini`/`zcode`) |
| `--lang LANG` | `python` | язык проекта — прописывается в `.serena/project.yml` |
| `--comm-lang LANG` | `en` | язык, на котором агент говорит с вами (строка в `CLAUDE.md`) |
| `--force` | — | перезаписывать уже существующие файлы |
| `--no-serena` | — | пропустить настройку Serena MCP |

Что делает `init` по шагам: копирует набор файлов (пропуская существующие без `--force`) →
проставляет язык проекта и язык общения → выставляет `default_agent` → дополняет `.gitignore` →
для не-Claude инструментов генерирует адаптеры (`.codex/prompts`, `.cursor/rules`, `.gemini/prompts`) →
для claude/zcode вписывает SessionStart-хук в `.claude/settings.json` → настраивает Serena (если не `--no-serena`).

### Скиллы Claude Code (набираются в чате)

| Скилл | Делегирует | Результат |
|---|---|---|
| `/new-prd [грубая идея]` | субагенту **prd-author** | интервью по темам → PRD `ai-flow/docs/specs/PRD.md` (видение, пользователи, сценарии, ограничения, архитектура) |
| `/new-task <описание или ссылка на спеку>` | субагенту **task-author** | новая фича-папка в `ai-flow/docs/tasks/` с DesignReview и файлами задач |
| `/sync-docs` | субагенту **doc-keeper** | наполнение/обновление памяти `.serena/memories/*`, спеков и таблицы в `CLAUDE.md` |

Скиллы `/new-task` и `/sync-docs` намеренно «тонкие» — вся логика в субагентах. `/new-prd` ведёт
**интерактивное** интервью прямо в чате (субагент не может задавать вопросы пользователю). Вне Claude
Code используйте промты напрямую: `PROMT_PRD.md` вместо `/new-prd`, `PROMT_TASKS.md` вместо `/new-task`,
`PROMT_SERENA.md` вместо `/sync-docs`.

## Как устроен оркестратор (под капотом)

Понимание внутренностей помогает отлаживать. Вот что делает `run_tasks.py` на каждом цикле:

1. **Находит задачи.** Сканирует `ai-flow/docs/tasks/`, берёт каждую фича-папку (кроме скрытых),
   внутри — все `*.md`, кроме `README.md`. Порядок — по имени файла (префикс-таймстамп = хронология).
2. **Определяет выполненные.** Задачи, лежащие в `<фича>/done/`, считаются готовыми.
3. **Проверяет зависимости.** Строка `Depends on:` / `Зависит от:` в файле задачи ссылается на другие
   задачи по имени/подстроке. Задача «готова к запуску», только если все её зависимости уже в `done/`.
4. **Собирает промт** из: `PROMT_AGENT.md` + `CLAUDE.md` + указателей на спеки и память (по умолчанию
   именно указатели, не полный текст — агент дочитает нужное сам) + DesignReview фичи + последнего
   `CHANGELOG.md` + текста самой задачи + напоминаний.
5. **Запускает агента.** Команда берётся из `agents.yml` (например,
   `claude --model sonnet --print --dangerously-skip-permissions`). Промт передаётся через stdin
   (временный файл), вывод стримится в консоль. Рабочая директория — корень проекта, так что агент
   видит весь код, а не только `ai-flow/`.
6. **Определяет успех.** Успех = в выводе есть маркер `<promise>COMPLETE</promise>` **или** агент вышел
   с кодом 0. Иначе — провал.
7. **Фиксирует результат.** При успехе: **переносит** файл задачи в `<фича>/done/` и (если это git-репо
   и коммиты включены) делает коммит одной строкой `feat: task #{id} - {title}`, обрезая до 155 символов,
   без упоминаний ИИ/инструментов.
8. **Обрабатывает сбои.** Три провала подряд — остановка. Тайм-аут задачи (по умолчанию 1200 с) —
   провал этой задачи.

> Важные нюансы: агенту **запрещено** самому делать `git commit` — это делает оркестратор. Файл задачи в
> `done/` переносит тоже оркестратор, а не агент. Маркер завершения агент печатает сам по инструкции из
> `PROMT_AGENT.md`.

## Модель задач и документации

**Задачи группируются по доработкам.** Одна папка = одна доработка (фича или фикс — не важно):

```
ai-flow/docs/tasks/<YYYYMMddHHmm_FEATURE_NAME>/
├── README.md                          # DesignReview: суть, ценность, архитектура, критерии
├── <YYYYMMddHHmm_TASK_SUMMARY>.md      # один файл = одна атомарная задача
└── done/                              # сюда run_tasks.py переносит выполненные (не создавайте вручную)
```

- `YYYYMMddHHmm` — таймстамп. Получить: `date +%Y%m%d%H%M` (Linux/macOS) или
  `Get-Date -Format yyyyMMddHHmm` (PowerShell на Windows).
- `FEATURE_NAME` / `TASK_SUMMARY` — kebab-case; сводка задачи ≤ 5 английских слов.
- Порядок выполнения = порядок имён; `Depends on:` уточняет зависимости.
- Заголовок задачи = первый `#`-заголовок в файле (по нему парсер берёт название).
- `README.md` внутри фича-папки задачей **не считается** — это её DesignReview.

**Спецификации и функциональность** живут вместе в `ai-flow/docs/specs/` (без отдельной папки
`functionality/`):

- корневой `README.md` — обзор проекта (заполняется при онбординге, разделы Overview/Capabilities);
- по папке на реализованную фичу: `README.md` — **целевое** поведение (что должно быть),
  `IMPLEMENTED.md` — **как реализовано** по факту. Обновляются автоматически после задач.

**Журнал** `ai-flow/docs/CHANGELOG.md` — только хронология («что и когда сделано»).
Переиспользуемые паттерны в журнал **не** пишутся — они идут в память Serena (единое хранилище знаний).

Полные шаблоны DesignReview и файла задачи — в `ai-flow/docs/tasks/README.md` и `PROMT_TASKS.md`.

## Конфигурация: agents.yml

`ai-flow/agents.yml` настраивает и оркестратор, и доступных агентов. Разбор по полям:

```yaml
default_agent: claude          # агент по умолчанию (если не передан --agent)
task_timeout: 1200             # тайм-аут одной задачи, секунд
completion_marker: "<promise>COMPLETE</promise>"   # что печатает агент по завершении

git:
  enabled: true                # false → не коммитить; авто-пропуск, если это не git-репо
  auto_commit: true
  message_template: "feat: task #{id} - {title}"   # обрезается до ≤155 символов, без упоминаний ИИ

context:                       # все пути — относительно корня проекта
  tasks_dir: ai-flow/docs/tasks
  specs_dir: ai-flow/docs/specs
  memories_dir: .serena/memories
  changelog_file: ai-flow/docs/CHANGELOG.md
  agent_prompt: ai-flow/docs/prompts/PROMT_AGENT.md
  inline_memories: false       # false → подкладывать указатели на память; true → полный текст

agents:
  claude:                      # Claude Code
    command: "claude --model {model} --print --dangerously-skip-permissions"
    model: sonnet
  codex:                       # OpenAI Codex CLI (промт читается из stdin через хвостовой "-")
    command: "codex exec -m {model} --dangerously-bypass-approvals-and-sandbox --skip-git-repo-check -"
    model: gpt-5-codex
  zcode:                       # zcode / Z.ai GLM (headless)
    command: "zcode -p -m {model}"
    model: glm-4.6
    # env:                     # для Claude-совместимого прокси
    #   ANTHROPIC_BASE_URL: "https://api.z.ai/api/anthropic"
    #   ANTHROPIC_AUTH_TOKEN: "${ZAI_API_KEY}"
```

- `{model}` в строке `command` подставляется из `model` агента или флага `--model`.
- Флаги codex/zcode **сверяйте с их `--help`** — реальные ключи могут отличаться; меняется только строка
  `command`, остальной флоу не зависит от инструмента.
- `inline_memories: true` полезно для агентов без доступа к Serena MCP — тогда весь текст памяти
  вставляется прямо в промт (дороже по токенам, но не требует MCP).

## Память проекта (Serena)

**Что это.** [Serena](https://github.com/oraios/serena) — MCP-сервер, который даёт агенту умный поиск и
навигацию по коду (символы, ссылки) и удобную работу с *памятью* проекта. Память — это обычные `.md`-файлы
в `.serena/memories/`, поэтому она **читается напрямую даже без MCP**; MCP нужен лишь для удобства.

**Что уже поставляется** (не удалять, только расширять):

- `suggested-commands` — какие команды флоу запускать и что где лежит;
- `task-completion` — критерий «задача выполнена» (сборка, тесты, журнал, обновление доков).

**Как наполнять память проекта.** Запустите онбординг — субагент `doc-keeper` по `PROMT_SERENA.md`:
изучит код, выделит устойчивые паттерны (архитектура, тесты, доступ к БД, сборка…), запишет их
как память и пропишет строки `read_memory(...)` в таблицу «Project knowledge» в `CLAUDE.md`.

```
/sync-docs
```

**Как настроить Serena MCP.** Автоматически:

```bash
python ai-flow/init.py setup-serena --tool claude    # или на этапе init
```

Требуется `uv`/`uvx`. Ручной fallback для Claude Code:

```bash
claude mcp add serena -- uvx --from git+https://github.com/oraios/serena \
  serena start-mcp-server --context ide-assistant --project "$(pwd)"
```

Для Codex — секция `[mcp_servers.serena]` в `~/.codex/config.toml`; для Cursor — `.cursor/mcp.json`
(обе `init.py setup-serena` создаёт сам). Документация: <https://github.com/oraios/serena>.

## Хук синхронизации памяти

`ai-flow/hooks/check_memory_sync.py` следит, чтобы таблица `read_memory(...)` в `CLAUDE.md` и реальные
файлы в `.serena/memories/` не разъезжались, и предупреждает о:

- **висячих строках** — таблица ссылается на память, которой нет;
- **сиротах** — файл памяти есть, но в таблице на него нет строки.

Это **best-effort напоминание** (не блокирует), запускается как `SessionStart`-хук — при старте сессии.
`init.py` подмешивает его в существующий `.claude/settings.json`, не затирая другие хуки (для инструментов
claude/zcode).

Настройки — `ai-flow/hooks/hooks.config.json`, секция `memory_sync`:

| Ключ | Значение | Смысл |
|---|---|---|
| `enabled` | `true`/`false` | главный выключатель |
| `mode` | `"warn"`/`"off"` | warn — печатает уведомление; off — молчит |
| `check_dangling` | `true`/`false` | сообщать про строки таблицы без файла памяти |
| `check_orphans` | `true`/`false` | сообщать про файлы памяти без строки в таблице |
| `ignore_memories` | список | какие записи не считать «сиротами» (по умолчанию поставляемые) |

Полностью отключить — `mode: "off"` или убрать блок `SessionStart` из `.claude/settings.json`.
Сделать блокирующим — зарегистрировать на gating-событие и возвращать `exit(2)` при проблемах.

> **Три уровня «строгости» инструкций** полезно различать: комментарий в `CLAUDE.md` — пассивная заметка;
> правило-буллет — агент следует ему по мере работы; **хук — единственное детерминированное «автоматически»**
> (запускает сама среда, а не агент).

## Языки

Различают два языка:

- **Язык артефактов флоу** (промты, субагенты, скиллы, память) — всегда **English**. Так шаблон остаётся
  универсальным и переносимым между инструментами.
- **Язык общения** — как агент разговаривает с вами. Задаётся `init.py --comm-lang` и живёт строкой
  `**Communication language: …**` в `CLAUDE.md`. Меняется в любой момент правкой этой строки.

Язык самого проекта (для навигации Serena) задаётся `--lang` и пишется в `.serena/project.yml`.

## Не коммитить флоу в git

Если не нужно тащить флоу в репозиторий, исключите весь набор (`ai-flow/`, `.claude/`, `.serena/`,
`CLAUDE.md`, `AGENTS.md`) через **глобальный gitignore** (личный, на все репозитории) — тогда
репозиторный `.gitignore` не трогается:

```bash
git config --global core.excludesFile ~/.gitignore_global
printf '%s\n' 'ai-flow/' '.claude/' '.serena/' 'CLAUDE.md' 'AGENTS.md' 'BUILD_PROMPT.md' >> ~/.gitignore_global
```

Для одного репозитория без коммита правила — те же строки в `.git/info/exclude`. Помните: `.gitignore`
действует только на **неотслеживаемые** файлы; уже закоммиченное сначала уберите из индекса:
`git rm --cached <path>`.

## FAQ и решение проблем

**«No tasks in ai-flow/docs/tasks».** Нет ни одной фича-папки с задачами. Создайте доработку через
`/new-task` (или вручную по `PROMT_TASKS.md`). `README.md` внутри фича-папки задачей не считается.

**«ERROR: PyYAML is required».** Поставьте зависимость оркестратора: `pip install pyyaml`.

**Оркестратор не коммитит.** Проверьте, что вы в git-репозитории (`git status`) и что в `agents.yml`
`git.enabled` и `git.auto_commit` = `true`. Вне git-репо коммиты просто пропускаются — это нормально.

**«No tasks ready (unmet dependencies)».** У всех оставшихся задач есть незакрытые `Depends on:`.
Проверьте, что имена в `Depends on:` совпадают (по подстроке) с именами файлов задач, от которых зависите,
